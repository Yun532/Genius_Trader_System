# PokieTicker A 股改造项目记忆

## 项目定位

本项目目标是把 PokieTicker 改造成一个面向 A 股的个股研究工作台。

核心不是做实时交易系统，也不是完整券商级基本面平台，而是保留原项目的主要价值：

- 个股 K 线展示
- 新闻、公告、财报等事件叠加到 K 线上
- AI 辅助解释某段上涨或下跌
- 历史相似事件回看
- 短期走势影响的实验性分析

第一阶段应定位为“个股事件复盘 + 新闻公告归因 + 财报影响分析”的研究工具。

## 原项目可复用方向

原 PokieTicker 的核心形态适合保留：

- 前端展示 K 线，并在时间轴上标记新闻事件
- 后端存储行情、新闻、事件、情绪分析结果
- 使用 LLM 对新闻和事件进行解释、分类、情绪判断
- 使用历史数据寻找类似事件或类似走势
- 支持用户围绕单只股票做研究，而不是先做全市场选股

## A 股改造重点

A 股版本需要重点替换和增强以下部分：

### 数据源

- 使用 AKShare 或其他 A 股数据源获取日线行情
- 支持 A 股代码体系，如 `000001`、`600519`、创业板、科创板、北交所等
- 明确复权口径，优先支持前复权、后复权、不复权的可选配置
- 增加 A 股交易日历、节假日、停牌、涨跌停等市场特性

### 事件体系

除新闻外，应增加 A 股更重要的事件来源：

- 公司公告
- 季报、半年报、年报
- 业绩预告、业绩快报
- 减持、增持、回购
- 定增、重组、并购
- 监管问询、处罚、立案调查
- 龙虎榜、异常波动公告
- 政策、行业、宏观相关事件

## 财报信息

财报应作为独立结构化数据源，不应只依赖新闻覆盖。

原因：

- 新闻对财报覆盖不完整
- 新闻是二手表达，可能有选择性和滞后
- 财报本身可以结构化，适合长期积累和量化比较

需要重点记录：

- 公告日期
- 报告期
- 营业收入
- 归母净利润
- 扣非净利润
- 经营现金流
- 毛利率
- ROE
- 同比变化
- 环比变化
- 是否超预期或明显低于预期

分析股价影响时，应优先使用“公告日期”，因为市场反应通常发生在披露后，而不是报告期结束日。

## K 线更新频率

第一版建议做日线级研究，不追求盘中实时交易。

推荐节奏：

- 日 K：每天收盘后更新一次，例如 15:30 到 17:00 之后
- 新闻和公告：每天多次更新
- 分钟线或实时行情：作为第二阶段能力，不作为第一阶段核心目标

原因：

- 免费数据源的实时稳定性、延迟、限频不一定可靠
- 事件研究和财报分析更适合日线级别
- 先做稳定研究台，比一开始做实时行情系统更容易落地

## 存储需求

第一阶段存储压力不大。

大致估计：

- 全 A 股 10 年日 K：几 GB 级别
- 财报结构化数据：通常几百 MB 到数 GB
- 新闻、公告摘要和元数据：几 GB 到十几 GB
- 如果保存公告 PDF、新闻全文、向量索引，可能增长到几十 GB

第一版可以使用 SQLite 或 PostgreSQL。

建议：

- 本地快速开发可以先用 SQLite
- 如果后续做长期服务和多用户，再迁移 PostgreSQL
- 新闻全文和公告原文是否长期保存，应作为配置项

## LLM 使用原则

LLM 不是用来替代数据源，而是用来整理和解释数据。

适合 LLM 做的事情：

- 新闻摘要
- 公告摘要
- 财报重点提取
- 事件类型分类
- 利好、利空、中性判断
- 影响逻辑解释
- 某段行情上涨或下跌的归因
- 历史相似事件的语义匹配
- 生成面向用户的研究摘要

不应让 LLM 凭记忆直接判断 A 股。

所有分析都应尽量基于系统提供的行情、公告、财报和新闻上下文。

## DeepSeek 与 OpenAI 选择

推荐第一版主力使用 DeepSeek，关键结论或高质量报告可接入 OpenAI 作为兜底或增强。

原因：

- A 股数据主要是中文新闻、公告、财报，DeepSeek 处理中文材料性价比高
- 项目会有大量批处理任务，例如新闻分类、公告摘要、财报提取，成本敏感
- OpenAI 可作为复杂归因、最终报告、质量复核的高质量选项

建议设计成可插拔模型层：

```env
LLM_PROVIDER=deepseek
LLM_MODEL=deepseek-v4-flash

LLM_FALLBACK_PROVIDER=openai
LLM_FALLBACK_MODEL=gpt-5.4-mini
```

后续代码实现时，模型调用不要写死。

## 推荐架构

建议分层：

- 数据层：AKShare、公告、新闻、财报
- 事件层：把新闻、公告、财报统一转成标准事件
- 分析层：规则、技术指标、财报指标、LLM 解释
- 模型层：DeepSeek、OpenAI 或其他模型可切换
- 前端层：K 线图上叠加事件点，点击查看摘要、解释、原文链接和相关财报

## 第一阶段目标

第一阶段优先做一个可用的单股研究闭环：

1. 输入 A 股代码，展示日 K 线
2. 拉取并展示相关新闻、公告、财报事件
3. 把事件标记到 K 线对应日期
4. 点击事件可查看摘要、类型、利好利空判断
5. 选择一段行情，让 LLM 基于事件解释上涨或下跌原因
6. 支持基础历史事件回看
7. 点击某一根 K 线或事件点，展示“当日涨跌原因”：OHLC、涨跌幅、当日/前一交易日附近事件、本地规则归因；有 LLM key 时再由 DeepSeek/OpenAI 基于系统上下文整理中文解释。

同步可靠性是第一阶段体验的一部分：`/api/stocks/{symbol}/sync` 需要返回日 K、新闻、公告、财报的分源状态、数量、失败原因和是否使用缓存。任何单一事件源失败都不应拖垮整个同步；日 K 失败时若有缓存，前端继续展示旧行情并提示“行情使用缓存”。

## 第二阶段目标

第二阶段再增强：

1. 加入财报结构化对比
2. 加入行业或同类公司对比
3. 加入龙虎榜、资金流、北向资金等 A 股特色数据
4. 加入向量检索，寻找语义相似事件
5. 加入短期影响预测实验
6. 支持多股票 watchlist

## 2026-05-14 进展：每日归因加入市场背景

当前 `daily-reason` 已不只看个股事件，还会纳入大盘和资金背景：

- 新增三大指数日线缓存：上证指数、深证成指、创业板指，独立存入 `market_index_ohlc`。
- 按股票代码自动匹配主要参考指数：沪市主板看上证，深市主板看深证成指，创业板看创业板指。
- 每日归因返回 `market_context`，包括指数涨跌幅、个股涨跌幅、超额收益、跟随大盘/逆势/跑赢跑输标签。
- 每日归因返回 `fund_context`，优先使用北向资金最近交易日净流入/流出，作为市场资金情绪背景。
- 前端“当日涨跌原因”面板新增“市场背景”，无指数或资金数据时降级提示，不影响行情和事件展示。
- LLM prompt 已加入市场/资金背景，但仍要求只基于系统提供的数据解释，不能凭记忆补 A 股事实。

限制仍然明确：这仍是日线级可能归因，不是分钟级因果解释；行业指数和板块资金还没有作为强依赖，后续可以作为第二层增强。

## 2026-05-14 进展：行业背景与行情后备源

每日归因继续增强为“个股 + 大盘 + 行业 + 资金 + 事件”：

- 个股日 K 主接口失败时，新增 `stock_zh_a_daily` 后备源；`600519` 这类主接口偶发失败的股票可以继续尝试拿到日 K。
- 新增个股行业映射缓存 `stock_industry_map`，优先从 `stock_individual_info_em` 获取行业字段。
- 新增行业板块日 K 缓存 `industry_board_ohlc`，用于计算行业涨跌幅、个股相对行业超额收益、跟随行业/逆板块/跑赢跑输行业标签。
- `daily-reason` 新增 `industry_context`，前端“市场背景”展示大盘指数、北向资金、行业板块三类信息。
- 行业接口不稳定时必须降级为“暂无行业板块背景”，不影响个股行情、公告、财报和本地归因。

新闻状态也要明确：当前新闻是 AKShare 东方财富个股新闻接口提供的真实新闻标题、时间、来源、链接和可用摘要，不是 LLM 编造；但还没有做新闻正文深抓、全文清洗、跨源去重和长期全文归档。

本轮围绕实际使用体验做了修正：

- 默认同步窗口改为最近 2 年到最近可获得交易日，`000001`、`600519` 均已验证不再停留在 2025 年 1 月测试窗口。
- K 线横轴已改成交易日序号轴，横轴标签仍显示真实日期，周末和节假日不再拉开空档。
- 新增 `daily_reason_cache` 和批量接口 `/api/stocks/{symbol}/daily-reasons`，单日归因优先读缓存；图表加载后会后台预热最近一批本地归因。
- 右侧“当日涨跌原因”调整为更清晰的研究顺序：当天新增新闻、市场背景、关联事件影响、可能归因、证据线索、利好/利空因素。
- 点击某根 K 线后，顶部 header 继续保留当天 OHLC、涨跌幅等交易信息，右侧面板专注展示新闻、背景和归因。
- 右侧原本空置的 `prediction-area` 已改为“研究面板”，用于承载每日归因、区间事件和区间归因；底部左侧保留事件列表和事件筛选。
- 选中某一天或点击某个事件后，底部资讯类型按钮和“全部/利好/利空”按钮只统计当天事件；未选日期时才显示全量统计。这样用户看单日归因时，筛选数量和当天新闻/公告/财报上下文保持一致。
- 股票展示增加名称优先策略：后端 `tickers.name` 会在名称为空或仍等于代码时自动用 AKShare/本地 fallback 补齐；前端下拉、同步详情、每日归因和区间归因标题使用“股票名称（代码）”，归因正文优先用股票名称，避免只显示代码。
- 已同步股票列表增加批量更新能力：新增 `POST /api/stocks/sync-watchlist`，可按已添加股票逐只同步日 K、新闻、公告和财报；前端 header 新增“更新全部”和最新行情日期。后端也预留可选本地调度配置 `AUTO_SYNC_WATCHLIST`，默认关闭，开启后可在收盘后按配置时间每日更新 watchlist。
- 右上角外部 GitHub 链接已从运行界面移除，避免干扰研究工具主界面。
- 搜索增加本地 fallback：当 AKShare 全市场股票列表接口断连或返回空时，仍可搜索本地兜底股票和已添加股票。当前已加入并同步 `sz002339` 积成电子，两年日 K、新闻、公告、财报和每日/区间归因可用。
- 外部资讯搜索进入第一版实现：新增 `backend/web_search.py`，支持 `NEWS_WEB_SEARCH_PROVIDER=openai|tavily`。OpenAI 走 Responses API Web Search，Tavily 走 Tavily Search API；二者都只负责找来源事实，搜索范围包括新闻、公告解读、业绩动态、行业事件、龙虎榜、游资席位、主力资金、资金流向、北向资金、融资融券等。结果统一入库为 `news` 或 `capital` 事件，再交给 DeepSeek/OpenAI 分析。新增 `POST /api/stocks/{symbol}/refresh-web-info` 和前端“搜索资讯”按钮；旧 `/refresh-web-news` 保留兼容。`NEWS_WEB_SEARCH_ENABLED=true` 时也会跟随单股同步自动搜索。
- 双模型分析进入第一版实现：`LLM_ANALYSIS_MODE=parallel_review` 时，DeepSeek 和 OpenAI 会各自基于同一份系统上下文独立做每日/区间归因，再优先由 OpenAI 汇总为最终结论，并返回 `model_consensus`、`model_disagreements` 和 `model_reviews`。前端每日归因和区间归因会显示“模型复核”小节。默认仍是 `primary_with_fallback`，避免增加成本和延迟。
- K 线支持鼠标滚轮缩放、Shift + 滚轮左右平移、双击重置，缩放后仍使用交易日序号轴定位事件点和点击日期。
- A 股颜色语义统一为红涨绿跌；图表 K 线、顶部涨跌幅、收益标签、利好/利空颜色都按 A 股习惯显示。
- 事件点改为按类型优先上色：新闻蓝色、公告黄色、财报青绿色，避免公告被默认蓝点混淆。

新闻源现状：

- 已接入真实 AKShare/东方财富个股新闻，不是模型生成新闻。
- 当前 AKShare `stock_news_em` 默认返回量偏少，常见只有最近 10 条；后续需要扩展多源新闻和全文抓取。
- 当天没有新新闻时，前端明确展示“当天新增新闻 0 条”，同时保留前一交易日至当日窗口内的公告、财报和新闻作为归因证据。

## 多模型与 Web Search 设想

可以把新闻增强和模型归因分成两层：

- 事实层：系统先用 AKShare、公告、财报、后续可选 Web Search 抓取真实新闻事实，并做去重、来源、发布时间、链接、摘要入库。
- 分析层：DeepSeek 和 OpenAI 只分析系统已经抓到并入库的事实，不凭记忆补新闻。

建议执行策略：

- 只配置 DeepSeek：DeepSeek 做新闻/公告/财报摘要、情绪分类、每日归因；新闻事实仍由系统数据源抓取。
- 只配置 OpenAI：OpenAI 做摘要、分类和归因；如启用 OpenAI Web Search，可作为额外新闻发现源，结果仍需入库去重后再分析。
- 同时配置 DeepSeek 和 OpenAI：DeepSeek 优先做中文批量摘要与初步归因，OpenAI 做独立复核和最终汇总；两边结论不一致时在前端标记“模型分歧”。
- 配置更多模型：统一走 provider 抽象，要求每个模型输出结构化 JSON，再由主汇总模型合并，不能让多个模型直接覆盖数据库事实。

重要限制：

- OpenAI Web Search 可以作为可选新闻发现源；DeepSeek 是否提供可用的 API 级 Web Search 需要按官方能力确认，不能假设一定存在。
- Web Search 搜到的新闻需要按 `title + source + published_at + url` 去重，并和 AKShare 新闻、公告、财报统一进入事件表。
- 第一版先不做自动全文爬虫，后续可以增加“LLM 复核/刷新当天新闻”按钮，对某天或某段区间重新抓取并整理。

## 暂不优先做的事情

以下内容不作为第一阶段重点：

- 高频实时交易
- 自动买卖信号
- 全市场复杂选股系统
- 完整三表财务建模
- 机构研报级深度估值系统
- 面向多用户的大型 SaaS 架构

## 产品判断

正确的第一版应该让用户能够回答：

- 这只股票最近为什么涨或跌？
- 哪些新闻、公告、财报可能影响了走势？
- 财报公布后市场如何反应？
- 类似事件过去发生过吗？
- AI 对这段行情的解释是否有证据支撑？

如果第一版能把这些问题回答清楚，就已经具备实际研究价值。

## 2026-05-15 进展：机构评级/投行预期板块

本轮新增“投行预期/机构评级”作为每日研究背景的一部分：

- 新增数据库表 `analyst_ratings` 和 `analyst_rating_sync`，缓存巨潮/AKShare 机构评级数据，字段包括发布日期、机构、分析师、投资评级、评级变化、前次评级、目标价上下限。
- 新增 `GET /api/stocks/{symbol}/analyst-ratings`，支持按时间窗口读取缓存并可刷新；同步 `POST /api/stocks/{symbol}/sync` 时也会尝试拉取近 180 天机构评级。
- `daily-reason` 新增 `analyst_context`，每日归因会展示最新评级、可用目标价中枢、相对当日收盘价的参考空间；若报告未披露目标价，明确显示“未披露目标价”，不让 LLM 编造。
- 前端“当日涨跌原因”面板新增“投行预期”小节，放在市场背景之后、关联事件之前；展示最近评级卡片、目标价状态、分析师和数据来源。
- 机构评级抓取隔离到子进程中执行。原因是 AKShare 的巨潮接口在部分 Windows 环境会触发 `py_mini_racer` 原生崩溃，子进程隔离后，即使上游崩溃也只会返回失败/缓存，不会拖垮 FastAPI 主服务。

限制仍然明确：巨潮接口很多报告只披露评级、不披露目标价；第一版只把它作为预期参考，不等同于券商完整研报估值，也不作为确定涨跌原因。

## 2026-05-15 进展：原版研究闭环保真与信号参考

本轮围绕原版 PokieTicker 的事件研究闭环继续补齐：

- 右侧固定为“研究面板”，按交互切换每日归因、事件深挖、相似事件、相似日期、区间归因和信号参考。
- 事件卡片新增“深挖”和“相似事件”入口；点击 K 线事件点会进入单事件深挖，而不是只打开每日归因。

## 2026-05-15 进展：LLM 成本控制与缓存策略

本轮加强了 LLM 调用控制，避免同一日期、同一区间或同一信号参考反复消耗 token：
- 新增通用 `analysis_cache` 表，用于缓存区间归因和信号参考的 LLM 结果，缓存 key 会包含股票、日期/区间、问题、模型配置和上下文摘要 hash，避免上下文变化后误用旧结论。
- 新增配置项：`LLM_CACHE_ENABLED`、`LLM_RECENT_AUTO_DAYS`、`LLM_DAILY_REASON_CACHE_TTL_HOURS`、`LLM_RANGE_ANALYSIS_CACHE_TTL_HOURS`、`LLM_SIGNAL_REFERENCE_CACHE_TTL_HOURS`，已写入 `.env.example`。
- 每日归因增加 `llm_policy`：默认只对最近 `LLM_RECENT_AUTO_DAYS` 天自动使用 LLM；更早的历史日期默认本地规则归因，除非 API 显式传 `force_llm=true`。
- 区间归因增加 TTL 缓存和 `use_llm/force_llm/refresh_cache` 参数；前端会显示“本地归因 / LLM 已使用 / LLM 结果来自缓存”。
- 信号参考增加 TTL 缓存，浏览器中已验证：第一次生成 LLM 情景推演，第二次打开同一上下文直接显示“LLM 结果来自缓存 · 缓存至 ...”。

当前默认策略是保守省 token：同步、批量预热、历史日期都不主动调用 LLM；只有近期、未命中缓存、且没有显式关闭 LLM 的研究请求才会触发模型。

补充：DeepSeek 请求不再传 `max_tokens` 上限，避免推理模型把过小 token 预算消耗在 reasoning 内容上导致正文为空；OpenAI/fallback 请求仍保留调用侧上限。
